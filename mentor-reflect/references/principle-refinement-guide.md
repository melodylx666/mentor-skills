# 通用原则提炼指南

> 从具体技术经验中提取可复用原则的方法论。这是 mentor-reflect 的核心能力——如果复盘不能产生可复用的原则，那就只是一次聊天。

---

## 什么是好的技术原则

好的技术原则具备以下特征：

| 特征 | 反面（鸡汤） | 正面（工程原则） |
|------|------------|----------------|
| **具体** | "要做好缓存设计" | "任何涉及批量数据的存储操作都必须做大 Key 检查" |
| **有边界** | "性能很重要" | "Redis 大 Key 在超过 100MB 时必须拆分，适用于高频读写场景，不适用于低频冷数据场景" |
| **可验证** | "代码质量要保证" | "核心链路的每一次变更必须有压测报告，TP99 恶化超过 5% 不允许上线" |
| **有出处** | "要做好容错" | "支付回调场景必须做幂等处理，因为我们在回调重复投递上踩过坑" |

## 原则提炼四步法

### 第一步：找关键决策

从用户的复盘描述中，找出那个**最关键的决策时刻**。

**怎么找**：
- "当时犹豫了很久" → 这里有纠结，说明有取舍，背后必有原则
- "当时快速做了决定" → 快速决策通常依赖直觉，直觉背后是未被言明的经验
- "事后看应该换一种做法" → 意识到更好的选择，说明有了新的认知

**示例**：
> 用户说："当时犹豫要不要重启 Redis，怕丢数据，犹豫了 5 分钟最后还是重启了。"
> → 关键决策：重启 vs. 不重启

### 第二步：问"为什么选这个？"

对关键决策追问，找到决策背后的**权衡逻辑**。

**追问链**：
```
为什么选 A 不选 B？
  → 因为 B 有风险 X
    → 为什么风险 X 不可接受？
      → 因为当时流量大，B 的操作时间窗口不可控
        → 所以原则是："故障恢复场景下，确定性 > 最优性"
```

### 第三步：抽象为"if-then"规则

把权衡逻辑转化为带条件的规则。

**公式**：
```
当 [场景] 时，优先 [做 A]，因为 [理由]；但注意 [边界条件]。
```

**示例**：
```
当线上故障需要快速恢复，且恢复手段的副作用可控时，
优先选择最快的恢复方案（如重启），
因为 MTTR 每多一分钟都在持续造成业务损失。
注意：如果恢复操作会导致数据丢失，则需先评估数据损失成本。
```

### 第四步：标注适用边界

每条原则必须回答三个问题：

```
✅ 什么场景下适用？
❌ 什么场景下不适用？
⚠️ 有什么前置条件或例外？
```

## 常见场景的原则提炼示例

### 场景一：线上故障处理

| 原始经验 | 提炼后的原则 |
|---------|------------|
| "先重启恢复，再定位根因" | **故障恢复优先于问题根治** — 适用：线上故障、大 Key 问题；不适用：非紧急的性能优化、数据一致性要求极高的场景 |
| "告警出来了要有人盯着" | **告警响应要有严格的升级机制** — 适用：线上监控、值班场景；不适用：非关键链路的 info 级别告警 |

### 场景二：技术选型

| 原始经验 | 提炼后的原则 |
|---------|------------|
| "最后还是用了 Redis" | **选型要基于核心约束排序** — 当性能 > 一致性 > 成本时，优先选内存型存储；反之则优先选关系型数据库 |
| "自研的轮子不如开源的" | **优先选择社区活跃的开源方案** — 适用：通用技术需求、团队规模 < 20 人；不适用：核心竞争优势场景、有特殊合规需求 |

### 场景三：代码设计

| 原始经验 | 提炼后的原则 |
|---------|------------|
| "这个接口忘了做幂等" | **所有写接口必须设计为幂等** — 适用：支付、回调、消息消费等可能重复调用的场景；不适用：纯查询接口、一次性操作 |
| "这段代码复用性太差了" | **一个函数只做一件事** — 适用：业务逻辑、工具函数；不适用：性能敏感的热点路径（允许适度内联） |

## 当用户提炼不出原则时的兜底方案

如果经过多轮引导，用户仍然无法提炼出通用原则，不要强行编造。至少可以产出以下**最小可行产出**之一：

### 1. 避坑清单条目

将用户的经验转化为一条"不要做什么"的规则，存入团队的 FAQ 或知识库。

```
[坑] 不要在优惠券计算逻辑中使用全量用户的 Hash Key
[后果] 800MB 大 Key 导致 Redis 阻塞，订单超时 40 分钟
[建议] 大 Key 拆分 + 上线前做 bigkeys 扫描
```

### 2. 监控/告警建议

将用户的痛点转化为可操作的可观测性改进建议。

```
[现象] 大 Key 导致 Redis CPU 100%，但没有对应告警
[建议] 增加 Redis 大 Key 监控：bigkeys 定时扫描 + 单个 Key 大小超过 100MB 自动告警
```

### 3. Checklist 检查项

将用户的经验封装为一个检查项，纳入上线检查清单或 CI/CD 门禁。

```
[检查项] 涉及缓存操作的变更，必须做大 Key 评估
[来源] 支付中心 Redis 大 Key 事故
```

### 兜底优先级

```
提炼出通用原则 → 最佳
避坑清单条目 → 可接受
监控/告警建议 → 可接受
Checklist 检查项 → 可接受
什么都不产出 → 底线:不允许
```

## 原则质量自检

每条提炼出的原则，检查：

- [ ] **具体**：一个工程师读到这条原则，知道具体该做什么/不该做什么
- [ ] **有边界**：标注了适用场景和不适用场景
- [ ] **有出处**：打上了这次复盘的经验标签（可选，但建议保留）
- [ ] **可传递**：团队成员看了之后，能直接用到自己的工作中
- [ ] **不过时**：不是针对某个特定版本的临时规则，而是经得起时间检验的判断框架
