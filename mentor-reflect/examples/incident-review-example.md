# 复盘报告：线上订单超时 P0 事故

> **复盘主题**：支付中心 Redis 集群宕机导致订单超时
> **复盘深度**：完整复盘

---

## 问题背景

### 时间线与场景

- **发生时间**：2026-05-18 14:30 - 15:10（持续 40 分钟）
- **项目/模块**：支付中心 / 订单服务
- **团队角色**：订单服务模块负责人
- **外部环境**：正值 618 大促压测阶段，流量为平时的 3 倍。前一天刚上线了新的优惠券计算逻辑。

### 问题描述

14:30 开始，线上订单超时率从 0.1% 飙升至 23%，大量用户无法完成支付。监控告警在 14:32 触发，但值班同学在 14:35 才确认告警。最终在 15:10 通过重启 Redis 集群恢复。

### 初始目标

- 订单成功率 ≥ 99.9%
- 支付响应时间 ≤ 2s
- 故障恢复时间（MTTR）≤ 15 分钟

---

## 问题痛点

### 关键卡点

1. **告警确认延迟** — 告警触发了但没有第一时间处理。值班同学以为是大促正常流量抖动，等了 3 分钟确认是真实问题。
2. **Redis 大 Key 定位困难** — 知道 Redis 有问题，但不确定是哪个 Key 导致的。因为新上线的优惠券计算逻辑在一个 Hash Key 中存储了全量用户的优惠券信息，该 Key 的大小达到 800MB。
3. **重启决策犹豫** — 确认是大 Key 问题后，担心重启 Redis 会导致数据丢失和雪崩，犹豫了 5 分钟才决定重启。

### 踩过的坑

1. **大 Key 没有监控** — 上线前完全没有检查 Redis Key 的大小。错误假设：优惠券数据量不大，不会成为瓶颈。
2. **值班 SOP 不清晰** — 告警响应没有严格的升级机制。错误假设：大促期间有足够人力监控。
3. **优惠券计算逻辑未做压测** — 只在单机环境测试了功能，没有用大促流量做压测。错误假设：优惠券逻辑简单，不会出性能问题。

### 决策时刻

- **关键决策**：是否重启 Redis 集群
- **决策依据**：观察到 Redis CPU 100%、内存 90%，并且 `keys *` 命令执行超时。判断大 Key 已经导致实例不可用，重启是唯一选择。
- **决策质量**：正确。事后复盘看，不会有更优方案——大 Key 删除也需要时间，而重启是最快恢复手段。

---

## 解决方案

### 最终方案

1. 14:35 确认告警后尝试排查
2. 14:45 定位到大 Key 问题（优惠券 Hash Key 800MB）
3. 14:55 决定重启 Redis 集群
4. 15:05 重启完成
5. 15:10 订单成功率恢复正常

### 方案要点

1. **重启 Redis 集群** — 最直接的恢复手段，虽然粗暴但有效
2. **临时降级优惠券计算** — 重启后关闭优惠券计算逻辑，先恢复核心支付流程
3. **事后拆分大 Key** — 将全量用户的 Hash Key 拆分为每个用户独立的 Key

### 方案对比

| 对比维度 | 重启 Redis | 在线删除大 Key | 主从切换 |
|---------|-----------|---------------|---------|
| 恢复速度 | 5 分钟 | 不确定（删除大 Key 耗时不可控） | 10 分钟 |
| 数据风险 | 无持久化丢失风险（RDB 备份存在） | 可能阻塞实例 | 低 |
| 可执行性 | 高，操作简单 | 低，bigkeys 命令也可能超时 | 中，需确认从库状态 |
| 最终选择 | **✓** | ✗ | ✗ |

### 落地效果

- **客观指标**：故障时长 40 分钟，影响订单约 2.3 万单。恢复后订单成功率回到 99.95%。
- **主观评价**：业务方反馈"处理速度可以接受，但希望下次更快"。团队内部复盘认为告警响应流程需要优化。

## 反思提炼

### 如果重来一次

上线前用全量数据做 Redis 大 Key 检查，而不是只做功能测试。告警响应流程加入 "5 分钟未确认自动升级" 机制。

### 可复用的通用原则

#### 技术原则

1. **任何涉及批量数据的存储操作都必须做大 Key 检查** — 适用场景：上线新的缓存逻辑、批量数据写入 Redis/Mongo 等；不适用场景：单条数据操作、已知数据量很小的场景
2. **大促期间的变更必须做全链路压测** — 适用场景：涉及核心链路的变更、有流量放大效应的场景；不适用场景：非核心链路、管理后台类的变更
3. **故障恢复优先于问题根治** — 适用场景：线上故障处理、大 Key 问题；不适用场景：非紧急的性能优化、代码重构

#### 过程原则

1. **告警响应要有严格的升级机制** — 告警发出后 2 分钟未确认 → 通知值班人；5 分钟未处理 → 通知技术负责人
2. **生产环境变更上线前要有检查清单** — 包括但不限于：容量评估、大 Key 检查、压测报告、回滚方案

#### 沟通/协作原则

1. **故障处理期间，指定一个人对外沟通** — 避免多人同时更新状态导致信息混乱

### 知识沉淀建议

- 将 Redis 大 Key 检查纳入上线检查清单，固定在 CI/CD 流水线中
- 告警响应 SOP 更新到团队知识库，并在下个迭代安排演练

### 给后来者的建议

> 不要假设"数据量不大就不会出问题"——每次上线前问自己三个问题：这个数据量上生产会膨胀多少倍？有没有兜底方案？回滚要多长时间？
