# 搜索策略参考

> 为 mentor-guide 技能提供标准化的搜索流程、关键词模板和结果评判标准。
> 目标：以最小搜索成本找到可信答案，避免无休止的搜索循环。

---

## 一、搜索关键词模板

按问题类型分类，每类提供 3-5 个模板。使用时根据实际场景组合填充。

### 1. Bug / 报错

| # | 模板 | 示例 |
|---|------|------|
| 1 | `"{精确报错信息}"` + `"原因"` / `"root cause"` | `"NullPointerException" + "root cause"` |
| 2 | `"{精确报错信息}"` + `"解决"` / `"solution"` | `"Error: Unable to access jarfile" + "解决"` |
| 3 | `"{框架/语言}"` + `"{报错关键词}"` + `"常见原因"` | `"Spring Boot" + "BeanDefinitionStoreException" + "常见原因"` |
| 4 | `"{报错信息的前 50-100 字符}"`（去参数化后的纯错误消息） | 去掉 IP、文件路径、时间戳等动态部分后搜索 |

## 二、搜索轮次控制

采用 **2 轮搜索协议**，每轮设定明确目标和终止条件，控制总搜索次数 ≤5 次。

### 搜索轮次总览

```
Round 1 ──→ 精确搜索 ──→ 命中 → ✅ 输出结果
   │                   未命中
   ↓
Round 2 ──→ 扩展搜索 ──→ 命中 → ✅ 输出结果
                          未命中 → 🔄 经验分析模式
```

---

### Round 1：精确搜索

**目标**：用最精准的关键词直接找到匹配结果。

| 参数 | 值 |
|------|----|
| 搜索次数上限 | **≤2 次**（换 2 组不同关键词） |
| 搜索范围 | 官方文档 / GitHub Issues / Stack Overflow |
| 关键词策略 | 使用精确报错信息、完整函数名、异常类全限定名 |
| 停止条件 | 命中 **≥1 个 A 级来源** 或 **≥2 个 B 级来源** 且答案一致 |
| 后续动作 | 满足条件 → 输出结果；不满足 → 进入 Round 2 |

**Round 1 关键词构建原则**：
- 优先使用错误日志中的唯一标识（异常类名、错误码、SQL 语句片段）
- 去除日志中的动态变量（IP、时间戳、用户名、文件路径）
- 如为中文搜索结果不理想，切换英文关键词重试（算 1 次搜索）

---

### Round 2：扩展/兜底搜索

**目标**：放宽关键词限制，扩大覆盖面，最后一次尝试。

| 参数 | 值 |
|------|----|
| 搜索次数上限 | **≤3 次** |
| 关键词策略 | 同义词替换、降级抽象层级、去关键修饰词 |
| 搜索范围 | 博客 / 知乎 / 技术社区 / Dev.to / 掘金 / V2EX |
| 停止条件 | 命中 **≥1 个 B 级来源** 或 **多个 C+ 级来源结论收敛** |
| 后续动作 | 满足条件 → 输出并标注信源等级；不满足 → 切换到经验分析模式 |

**Round 2 关键词变形技巧**：
- 降级抽象：`NullPointerException` → `NPE` → `空指针` → `对象为空`
- 同义替换：`性能优化` → `性能调优` → `performance tuning` → `优化经验`
- 去修饰词：`Spring Boot 2.7 启动慢` → `Spring Boot 启动慢`
- 拆分组合：将长错误消息拆为 2-3 个片段分别搜索

## 三、搜索结果筛选标准

### 1. 来源质量评级

| 等级 | 来源类型 | 参考标准 |
|------|----------|----------|
| **A** | 官方文档、官方 GitHub Repo、RFC 标准 | Apache / Spring / CNCF / IETF 等官方渠道 |
| **B+** | 知名博客（作者为 PMC / Committer）、Stack Overflow 高赞（100+）、知名技术大会分享 | 作者有领域背书，内容经过同行检验 |
| **B** | 普通技术博客、Stack Overflow 普通答案、GitHub Issue 官方回复、知乎高赞（50+） | 逻辑自洽，有代码/数据支撑 |
| **C+** | 个人博客、论坛讨论、低赞 Stack Overflow 回答、V2EX 高赞讨论 | 仅作参考，需交叉验证 |
| **C** | 未经验证的贴吧、AI 生成的未经核实内容、过期文章（>5 年且无更新） | 不直接采纳，仅作线索 |

### 2. 相关性评分

| 分数 | 匹配程度 | 说明 |
|------|----------|------|
| 10 | 直接症状匹配 | 报错信息/场景完全一致，同一框架同一版本 |
| 8 | 相关症状匹配 | 报错信息一致但版本/环境不同 |
| 6 | 同类问题匹配 | 同一类型错误但组件不同 |
| 4 | 一般主题匹配 | 相关主题但不直接解决问题 |
| 2 | 弱相关 | 仅关键词部分命中，场景差异大 |

> 综合评分 = 相关性分 × 来源等级权重（A=1.0, B+=0.8, B=0.6, C+=0.4, C=0.2）
> 综合评分 ≥ 6 分且来源等级 ≥ B 时，视为可靠结果。

### 3. 来源时效性规则

| 技术类别 | 最长可接受时效 | 说明 |
|----------|---------------|------|
| 框架/语言核心 | 2 年 | Java/Python/Go 核心 API 变更慢 |
| 云原生/中间件 | 1 年 | Kubernetes、Docker、Flink 等版本迭代快 |
| 前端/移动端 | 6-8 个月 | React/Vue/Flutter 等频繁发版 |
| AI/LLM | 3-6 个月 | 模型和框架月月更新 |
| 安全漏洞 | 即时（≤1 个月） | CVE 需最新信息 |

> 超出时效但来源等级为 A 或 B+ 的，仍可参考基本原理，但需标注"注意时效性"。

### 4. 交叉验证规则

| 条件 | 结论 |
|------|------|
| ≥2 个 B+ 级以上来源结论一致 | 置信度高，可直接采纳 |
| 1 个 A 级 + 1 个 B 级结论一致 | 置信度较高，可采纳 |
| ≥3 个 C+ 级来源结论一致 | 置信度中等，建议谨慎 |
| 各来源结论冲突 | 以 A 级/A 级来源为准；无官方来源时，选择最合理的方案并标注争议点 |

## 四、搜索终止与兜底规则

### 1. 成功终止

**条件**：找到 ≥2 个 B 级及以上来源且答案收敛（结论一致）。

**动作**：
- 停止搜索
- 整理结果开始输出
- 在回答中注明参考来源

### 2. 部分结果

**条件**：2 轮搜索完成后，仅有 C 级 / C+ 级来源可用。

**动作**：
- 在回答开头标注：**「搜索结果有限，以下为参考」**
- 输出中找到的部分线索和推理过程
- 给出建议的进一步排查方向（如：建议查阅某官方文档、建议复现后补充日志等）
- 不做出确定性结论

### 3. 完全失败

**条件**：2 轮搜索全部完成，未找到任何 ≥C+ 级来源。

**动作**：
- 标记 **「搜索未找到直接匹配结果，切换到经验分析模式」**
- 基于已有知识进行分析推理
- 给出问题分类和可能的原因范围
- 提供排查框架而非具体答案（如：建议分段隔离测试、建议启用更详细日志等）
- 明确告知用户：以下为经验推断，非搜索结果支撑

### 4. 强制终止边界

- **单问题的总搜索调用次数上限**：**5 次**（Round 1 ≤2 次 + Round 2 ≤3 次）
- **不要为了搜索而搜索** — Round 1 找到足够可信结果立即终止

## 五、决策树速查

```
遇到问题
  ↓
分析问题类型 → 选择关键词模板
  ↓
Round 1 精确搜索（≤2次）
  ├── 找到 ≥1 A 或 ≥2 B 级 → ✅ 成功终止
  └── 未满足条件 → Round 2
        ↓
Round 2 扩展/兜底搜索（≤3次）
  ├── 找到 ≥1 B 级 或 多个 C+ 收敛 → ⚠️ 输出并标注信源
  └── 无有效结果 → 🔄 切换到经验分析模式
```
